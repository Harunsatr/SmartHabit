Authentication Flow\n\n```\n1. User submits email + password at /login\n   ↓\n2. NextAuth CredentialsProvider validates against database\n   ↓\n3. JWT callback creates secure token with user data including ROLE\n   ↓\n4. Session callback enriches session with token data\n   ↓\n5. Login page fetches /api/auth/session to get role\n   ↓\n6. Based on role:\n   - ADMIN → redirect to /admin\n   - USER → redirect to /dashboard\n   ↓\n7. Middleware intercepts subsequent requests\n   ✓ /admin with ADMIN role → Allow\n   ✗ /admin with USER role → Redirect to /dashboard\n   ✓ /dashboard with any auth → Allow\n   ✗ /dashboard with no auth → Redirect to /login\n```\n\n---\n\n## 🧪 Testing Checklist\n\n- [ ] Admin can login and see `/admin` dashboard\n- [ ] User can login and see `/dashboard`\n- [ ] Non-admin cannot access `/admin` (auto-redirect)\n- [ ] Unauthenticated cannot access protected routes\n- [ ] Session persists after page refresh\n- [ ] Role properly shown in network tab `/api/auth/session`\n- [ ] Logout clears session\n- [ ] Invalid login credentials show error\n- [ ] Database seed users work correctly\n\n---\n\n## 🐛 Debugging Commands\n\n```bash\n# Check session in browser console\nfetch('/api/auth/session').then(r => r.json()).then(d => console.table(d.user))\n\n# Check auth cookies\n# DevTools → Application → Cookies → localhost → look for \"next-auth\"\n\n# Database inspection\nnpx prisma studio\n\n# Run tests\nnpm run dev\n\n# Show NextAuth logs\n# Check console output while npm run dev\n```\n\n---\n\n## 📝 Adding New Protected Features\n\n### For Protected Pages\n1. Create page as Server Component\n2. Call `getServerSession(authOptions)` at top\n3. Check if session exists, redirect if not\n4. For admin-only, also check `session.user.role === \"ADMIN\"`\n\n### For Protected API Routes  \n1. Import guard function: `import { guardAuth, guardAdmin } from \"@/lib/api-guard\"`\n2. Call guard at route start\n3. Check `guard.success` and return error if false\n4. Use `guard.data` to access user info\n\n### For Protected Client Components\n1. Use `useSession()` hook from NextAuth\n2. Check `status === \"authenticated\"`\n3. Access `session?.user?.role` for role checks\n4. Use middleware as backup protection\n\n---\n\n## ⚙️ Configuration Files\n\n**Core Auth Files:**\n- `lib/auth/auth.config.ts` - NextAuth configuration\n- `lib/auth/auth.types.ts` - Type definitions\n- `lib/auth/auth.server.ts` - Server-side helpers\n- `lib/auth/auth.utils.ts` - Utility functions\n- `lib/api-guard.ts` - API protection utilities\n\n**Route Protection:**\n- `middleware.ts` - Global route protection\n- `app/admin/layout.tsx` - Admin dashboard layout\n- `app/(auth)/login/page.tsx` - Login form\n- `app/api/auth/[...nextauth]/route.ts` - NextAuth endpoint\n\n---\n\n## 🎯 Key Takeaways\n\n1. **Server-side is safer** - Use Server Components for auth checks when possible\n2. **Middleware as backup** - Routes are protected at multiple levels\n3. **Role in token** - User role is securely stored in JWT\n4. **Consistent patterns** - Use guard functions for all API routes\n5. **Test thoroughly** - Auth issues affect user experience significantly\n\n---\n\n**Last Updated**: March 2026\n**Status**: ✅ Production Ready\n
